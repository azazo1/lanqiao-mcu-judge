# 编码规范

## 仿真代码编写

新添加的模块应该使用独立的 rs 文件在 rust 模块中添加, 功能解耦, 方便拓展.

rust 代码编写的时候不要堆在一块, 也不要散落在同一个层级, 注重代码模块层级的设计.

编写 rust 仿真代码的时候, 不要基于 sample 中的模块使用设计来设计仿真代码, 比如 pca 和 超声波 的耦合, 其他人可能设计代码的时候, 超声波使用 timer0/1/2 而不是使用 pca 导致功能无法正常仿真.

构建外设的时候, 不要以上帝视角来编写代码, 要假设你开发的就是外设本身, 这个外设自身接收信号, 根据信号做出反应, 除此之外不会接收到其他的数据. 并且需要仔细阅读指定外设的手册, 尽量仿真手册中的各种参数, 而不是理想化的模型.

cpu 仿真: 这个 cpu 不是普通的 8051 单片机, 而是 stc15f2k60s2, 具体的定时器, 中断, 寄存器等内容, 需要阅读项目中的用户手册来获取准确的信息, 而不是使用 8051 芯片的通用知识.

仿真实现应具有一定的真实性, 不能假设所有程序代码都按照规定好的路线执行, 比如中断触发之后, 程序可能不清除中断标志位, 那么可能导致重复触发中断, 这个分支情况也是要兜住的.

## 测试运行

进行修改之后, 不要自动运行所有的评测脚本和测试, 因为他们太耗时间了, 你需要给出提醒, 可能的回归等内容, 当用户主动要求你执行评测的时候才执行.

测试运行的时候, 尽量使用 `--release`, 不然耗时太久了, 不过不需要直接使用 `./target/releaase/xxx` 的方式执行, 防止代码和可执行程序不对应, 直接使用基于 `cargo run --release -- ...` 的命令即可.

如果你觉得评测脚本运行过久, 可以使用 `RUST_LOG=debug` 逐句查看语句执行情况, 以免抓瞎.

测试运行脚本的时候 ckpt 的 INFO 输出不要屏蔽, 方便追踪.

## 评测脚本相关

评测脚本编写的时候, 对指定的测量量留有余量而不是全部都使用严格相等判断, 防止误差.
读取 seg 的时候, 可以将其中的数值提取出来, 判断是否符合指定的范围.

> [!note]
> 编写评测脚本的时候, 优先使用 rhai 已经有的特性, 尽量不要自己再新增一个接口.

修改评测脚本的接口定义需要同步更改 [docs/judge-script-manual.md](docs/judge-script-manual.md).

对于指定的真题评测, 在题目要求 pdf 文件中有性能要求的, 需要在设计评测脚本的时候遵循性能要求, 比如 relay 在多少多少毫秒之内作出反应, 那么就不要留有过大的空隙等待其反应.

编写评测脚本的时候, 不要根据当前的 hex 输出什么, 就觉得 expect 是什么, 而是要根据推导, 得出正确的答案来编写 assert, 用于判断 hex 是否正确. 不要使用 hex 的输出作为先验评测条目.
编写评测脚本 judge 和 README.md 应该充分阅读 sample 代码再编写, 应该充分了解 sample 代码的目标而非现有性质, 因为 sample 代码并不保证正确性, 评测脚本的目标就是从其中发现错误;
测试需要覆盖所有的边界条件 (BVA 使用三点而非两点), 运用软件测试的知识, 挑选覆盖准则, 分析测试需求, 并且编写测试用例评测脚本覆盖, 保证所有目标功能正确.
评测过程尽量可追溯, 而不是进行了一堆操作 (比如按键, uart 等等) 之后观察结果是否正确, 过程也同样重要.

评测脚本的 `run_to_*` 类函数超时不应设置过长, 根据实际情况而定比如 1s~2s 就已经足够, 再长说明功能实现有性能问题, 不过具体情况应该具体分析, 比如题目中又说 seg 在 100ms 内响应的话, 那么要改成 100ms 或者类似合理值.
评测脚本不要滥用 `run_to` 的回调功能, 其会严重影响性能.
不要使用 `run_to*` 做断言作用, 其应该仅用于相位对齐等功能.

在评测脚本中定义函数的时候, 如果这个函数涉及断言, 那么使用 `assert_` 开头命名, 不要使用 `wait_` 或者其他的前缀.

评测脚本的断言设计可以参考: [judge.md](samples/na16/judge.md), 这是 4t 原始的评测过程, 这个项目的目标就是完全达到 <https://www.4t.wiki/> 评测的效果.
评测脚本需要补充场景注释, 解释当前正在测评的目标, 操作的用意等信息, 方便用户快速确定其编写的 hex 文件针对题目存在什么问题.

评测脚本的注释和 assert 提示信息都使用中文编写, 不要使用英文.

编写评测脚本的时候可以使用 ckpt 函数, 整合一段操作, 用于判断评分点, 防止 assert 失败导致后面的内容都执行失败.
注意 ckpt 中, 最佳实践为先进行操作, 然后读取状态, 最后才是 assert, 防止 assert 失败导致中间状态的泄露, 如果你需要读取中间状态的值, 用于 assert, 那么可以先存储到局部变量当中, 在最后 assert.
编写检查点可以参考 [4t.rhai](samples/na16/judge/4t.rhai), 以及对应的 [judge.md](samples/na16/judge.md).

评测脚本断言优先利用已有的 assert_* 函数, 而不是直接使用 assert(...).

根据题目编写评测脚本的时候, 如果有参考的 4t 评测流程, 那么尽量减少使用 reset 来复位, 因为评测点之间的参数设置可能都是有联系的.

> [!important]
> 遵守一个重要规则: **编写评测脚本的时候不要阅读实现的源代码, 编写实现源代码的时候不要阅读评测脚本**.

## 出题

出题可以参考:

```
参考 na16 sample 的题目, 运用到 eeprom ds18b20 ds1302 key seg led pcf8591 uart1 ultrasound ne555 的一道综合运用各种底层的题目 <题目名, 比如 na17>, 主要是测试底层编写能力, 考察点要在考纲之内, 运用到尽量多的知识点:
eeprom: 读写, 锁(验证首次上电的数据是否有效).
ds18b20: 读温度, 设置温度精度
ds1302: 读写日期, 时间, 设置12/24(转换)
key: 单键短按, 双键长按, 双击,
seg: 数码管显示
led: led 闪烁 pwm; relay, motor, buzz 的控制.
pcf8591 AIN1 通道(光敏电阻)和 AIN3 通道(滑动变阻器) 的 ADC 和 输出 DAC
uart1 串口收发和复杂数据格式解析
ultrasound 可设置音速的距离测量
ne555 测量频率并运用频率数据
根据上述考点, 编写一道综合题, 并且编写在 sample 中 README.md 和完善的
judge 评测脚本,
```

出题的一个示例是 samples/na17, 出题请新建 samples 文件夹, 而不是直接在 na16 或者 la17 当中写.

ne555 要是出题就会和按键 S19 S18 S17 S16 冲突, 如果出题有串口, 那么就会和 S67 S10 S11 S14 S15 S18 S19 冲突, 这个编写到文档当中.

## justfile

编写 justfile 的时候, 需要同时考虑 unix 和 windows 两种操作系统的执行, recipe 执行需要等价,
除非明确声明某个 recipe 只能在某个平台中使用.

recipe 中的代码需要注重可读性而非健壮性, 明确其功能的边界, 不要弄的太复杂, 简介易读, 用于快速提供给用户快速调用成套命令的能力, 而不是作为脚本执行复杂任务.
