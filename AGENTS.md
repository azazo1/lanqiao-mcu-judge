# 编码规范

新添加的模块应该使用独立的 rs 文件在 rust 模块中添加, 功能解耦, 方便拓展.

你运行调试的时候添加 RUST_LOG=debug 环境变量方便观察, 但是不要写入到 README 等文件当中.

评测脚本编写的时候, 对指定的测量量留有余量而不是全部都使用 assert_eq_str, 防止误差.
读取 seg 的时候, 可以将其中的数值提取出来, 判断是否符合指定的范围.

评测脚本里的数值判断, 优先使用 Rhai 原生字符串切片配合 parse_int(...) 或 parse_float(...), 其是可以正确处理前缀零的, 不要把纯数值显示直接写成整串 display_text 比较.
如果需要判断空白位, 固定符号位, 时间分隔符等格式, 优先使用 Rhai 原生字符串切片, 例如 `s[0..5]`, 再配合 regex_is_match(...) 或 assert_eq_str(...).

对于固定范围的数字显示, 比如 8 位数码管的最后 5 位显示频率数字, 那么你可以直接提取最后五位的结果, 然后 parse_int, 而不是先判断 前缀空白/前缀0 然后判断后面的数值.
不过如果题目要求了是前缀空白或者前缀0, 那么你需要进行判断.

匹配单个字符, 尽量使用 `display_text()[0]` 这种索引, 而不是 `seg_pattern(1)`, 前者可读性更高, 后者更灵活, 仅在需要非字符的情况下用到后者.

> [!note]
> 编写评测脚本的时候, 优先使用 rhai 已经有的特性, 尽量不要自己再新增一个接口.

进行修改之后, 不要自动运行所有的评测脚本和测试, 因为他们太耗时间了, 你需要给出提醒, 可能的回归等内容, 当用户主动要求你执行评测的时候才执行.

修改 评测脚本 的内容需要同步更改 [docs/judge-script-manual.md](docs/judge-script-manual.md).

编写 rust 仿真代码的时候, 不要基于 sample 中的模块使用设计来设计仿真代码, 比如 pca 和 超声波 的耦合, 其他人可能设计代码的时候, 超声波使用 timer0/1/2 而不是使用 pca 导致功能无法正常仿真.
