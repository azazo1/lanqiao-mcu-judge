# 真题评测脚本编写 skill

## 适用场景

当需要为蓝桥杯单片机真题编写 `judge/*.rhai` 评测脚本时, 使用本流程. 适用对象包括:

- 根据题目 PDF, 4T 原始流程, `judge.md` (评测结果表格), 样例 hex 行为编写评测脚本.
- 将 4T 的检查点转成可执行断言.
- 优化已有脚本的评测时间, 减少重复 reset 和长等待.
- 修正脚本中为了迎合某个 hex 输出而产生的经验窗口.

重要边界:

- 写评测脚本时不要阅读被测题目的实现源码.
- 可以阅读题目 PDF, 图片页, `judge.md`, 参考真题脚本, 文档手册, 外设资料.
- 脚本应按题面要求和可推导模型判定, 不应以当前 hex 输出作为正确答案.

## 输入材料

优先收集这些材料:

- 题目 PDF, 包括图片, 表格, 波形, 接线图和性能要求.
- 同题目的 `judge.md`, 用作 4T 检查点顺序和展示内容参考.
- 相邻真题的 `judge/4t.rhai` 和 `judge.md`, 用作脚本组织方式参考.
- `docs/judge-script-manual.md`, 用作 Rhai 接口能力参考.
- 考纲, 知识点文档, 题目涉及外设的常见考法和边界条件.
- 必要的硬件手册和板级原理图.

不要只读取文字 OCR 结果. 如果 PDF 中有图, 需要查看页面图像, 尤其是显示格式, 接线关系, 传感器方向, 时序图和状态转移图.

## 编写流程

1. 先建立题面与考纲模型

   从 PDF 和考纲中整理所有可能考到的内容, 不只看题面显式流程. 至少覆盖外设使用, 页面含义, 按键行为, 串口协议, 传感器输入, 输出设备, 存储行为, 数据格式, 边界条件, 状态转移, 异常输入, 计算公式, 精度要求和性能上限. 对每个可评测要求写出对应的脚本断言类型.

2. 对齐 4T 检查点

   将 `judge.md` 的检查点编号, 测试条件, 正确结果逐项搬到 `ckpt(index, condition, expected, || { ... })`. 展示文本应和 `judge.md` 保持一致, 但不要复制 `我的结果` 作为断言依据.

3. 先写公共断言

   对常见对象建立专用函数:

   - `assert_*_page` 检查页面前缀, 格式和数值.
   - `assert_*_query` 检查串口响应格式和内容.
   - `assert_*_pattern` 检查 LED, relay, buzzer, motor 等输出状态.
   - `*_formula_window` 根据题面公式和误差要求生成容忍窗口.
   - `*_response_ns` 保存题面性能上限.

4. 再写场景推进

   将页面切换, 参数调整, 目的地设置, 启动运行, 等待状态, 重置等动作封装为小函数. 函数名要表达动作和断言边界. 如果函数内部有断言, 使用 `assert_` 前缀.

5. 按题面推导窗口

   对数值类断言优先从题面推导:

   - 速度页: 由公式, 输入量误差, 传感器精度, 显示舍入共同推导.
   - 距离页: 由超声波测距精度和显示舍入推导.
   - 坐标页: 由速度窗口, 行进时间, 目标路径长度, 采样相位和串口响应尾部时间推导.

6. 保留性能断言

   不要把题面性能要求改成简单的稳定等待. 对要求 `<=100ms` 或 `<=500ms` 的行为, 应使用 `run_to`, `run_to_event`, `run_to_ns` 等明确测量上限.

7. 优化执行时间

   初版可以让检查点独立复现, 但最终应尽量按 4T 流程连续复用状态. 对自然连续的检查点, 不要反复 reset 和重跑长路线. 对分叉时序不同的检查点, 保留独立启动.

## 考纲覆盖审计

建立模型时需要把题目要求和考纲可能考点放在一起审计. 即使 `judge.md` 没有显式列出, 也要判断是否需要补充边界或不变量检查.

常见审计维度:

- 基础 I/O: LED, relay, motor, buzzer 的默认状态, 状态切换, 保持行为和互斥关系.
- 按键: 单击, 多次按键, 长按, 切页, 参数选择, 参数上下界和回绕.
- 数码管: 页面前缀, 固定位格式, 空白位, 负号, 小数点, 舍入, 闪烁和刷新响应.
- 串口: 合法命令, 查询命令, 忙状态拒绝, 非法格式, 边界坐标, 回复格式和响应时间.
- 传感器: ADC 阈值, 超声波距离, NE555 频率, DS18B20 温度, DS1302 时间等输入精度和边界.
- 存储: EEPROM 首次上电, 有效标记, 参数保存, 掉电恢复和非法数据兜底.
- 计算: 公式输入范围, 舍入规则, 溢出风险, 负数处理, 路径积分和采样相位.
- 状态机: Idle, Busy, Wait, Alarm 等状态的进入条件, 退出条件, 输出联动和串口可见状态.
- 性能: 按键响应, 显示响应, 串口响应, 避障响应, 自动返回时间, 闪烁周期.
- 异常: 非法命令, 越界参数, 运行中修改目标, 障碍物未清除时启动, 传感器极值.

## 性能窗口建模

题面可能同时出现多个响应要求, 不要错误合并.

典型例子:

- 按键响应 `<=100ms`.
- 数码管响应 `<=100ms`.
- 如果测试的是按键后数码管显示指定值, 可以建模为 `按键响应窗口 + 数码管响应窗口`.
- 如果测试的是按键后 relay 或 LED 状态改变, 只使用按键到目标输出的对应窗口.
- 串口回复 `<=500ms` 应从发送命令后开始计时.
- 避障 `<=500ms` 应从距离跌破阈值后开始计时.

推荐结构:

```rhai
fn key_response_ns() {
    100_000_000
}

fn seg_response_ns() {
    100_000_000
}

fn key_to_seg_response_ns() {
    key_response_ns() + seg_response_ns()
}
```

## 数值窗口建模

速度类:

```text
v = 3.14 * R * F / 100 + B
```

如果题面给定频率测量误差 `+-8%`, 则使用 `F * 0.92` 和 `F * 1.08` 生成速度上下限. 显示一位小数时再加显示舍入窗口, 例如 `+-0.05`.

距离类:

如果题面给定测距精度 `+-3cm`, 则窗口为 `[distance - 3, distance + 3]`, 并按显示规则处理下界和舍入.

坐标类:

坐标不要直接写死当前 hex 的输出. 应按这些因素推导:

- 起点和终点坐标.
- 路径长度.
- 各段频率和持续时间.
- 速度公式误差窗口.
- 启动响应延迟.
- 串口查询响应延迟造成的尾部采样时间.
- 显示或串口坐标取整造成的边界误差.

## 检查点组织

`ckpt` 是显式评测点, 不是事务回滚点. 当前检查点失败后, 后面的检查点会继续执行, 但当前闭包失败后的语句不会继续执行.

因此, 连续状态复用时需要注意:

- 影响后续流程的动作尽量放在断言之前.
- 如果某个检查点的断言失败会导致后续状态不可用, 可以在下一个大段开头重新建立场景.
- 对重要时刻的状态可以先保存快照, 后续检查点只断言快照.
- 对不同时间线的检查点不要强行复用同一次运行.

适合连续复用:

- 初始空闲, 页面切换, 参数调整.
- 设置目的地, 启动, 运行中查询, 到达后检查.
- 等待态内的多个查询和显示页.
- 到达后继续修改参数, 重置, 设置新目标.

适合独立启动:

- 同一路线要求不同采样时刻, 但前一个检查点已经推进过该时刻.
- 同一目标要求不同变速时间线.
- 某个检查点会故意发送非法命令或破坏状态.
- 某个检查点验证上电默认值或掉电恢复.

## 可追溯性

每个检查点应让失败信息尽量指出原因:

- 页面前缀错误.
- 显示格式错误.
- 数值超出窗口.
- 串口格式错误.
- 串口响应超时.
- 继电器或 LED 状态错误.
- 坐标 X 或 Y 单独错误.

推荐让断言消息包含 `actual`, `expected`, `display`, `reply` 等现场信息.

## 时间优化方法

脚本慢通常不是因为 `100ms` 或 `500ms` 响应窗口, 而是重复长场景.

优化步骤:

1. 搜索 `run_ms`, `run_to_elapsed_from`, `reset`, `boot_scene`, `prepare_*`.
2. 找出 `run_ms(10000)`, `run_ms(11000)`, `run_ms(5000)` 这类长等待.
3. 判断这些等待是否只是为了把场景跑到某个后续状态.
4. 如果多个检查点本来就是连续流程, 用跨检查点变量保存启动时间, 停止时间, 快照.
5. 删除不再使用的旧准备函数, 避免误导维护者.
6. 最后估算新旧仿真时间, 确认优化幅度.

可保存的跨检查点状态:

- 启动按键时间和响应时间.
- 等待态切换时间.
- 某一时刻 LED 快照.
- 某段变速流程的起点信息.

## 验证清单

提交前至少检查:

- `ckpt` 数量和编号是否与 `judge.md` 对齐.
- `condition` 和 `expected` 是否和 `judge.md` 的检查点内容一致.
- 是否阅读了 PDF 图片页.
- 是否没有读取被测实现源码 (编写评测脚本的时候不应读取实现的代码, 防止先验).
- 性能要求是否用上限断言表达, 而不是固定等待后观察.
- 数值窗口是否从题面公式和精度推导.
- 坐标窗口是否没有采用当前 hex 的实际输出.
- 是否没有残留无用的长等待准备函数.
- `git diff --check` 是否通过.
- 如有跨 `ckpt` 状态, 是否验证过 Rhai 闭包可以读写外层变量.

## na15 经验摘要

`na15` 这类路径运行题容易慢, 因为很多检查点会反复从上电跑到中间状态. 优化时可把流程拆成几段:

- 初始空闲和参数页连续测.
- `(108,0)` 路线连续测运行中, 到达后 relay, LED, 串口查询.
- `(148,0)` 路线保存 `2s` LED 快照, 然后继续等到 `5s`.
- `(148,148)` 路线连续测运行坐标, 手动等待态, 等待页, 避障, 恢复运行.
- 到达 `(148,148)` 后继续调参数, 重置, 设置 `(120,160)`.
- `990` 多段变速按时间线分支, 不强行合并全部检查点.

估算仿真时间时要区分两类数字:

- 静态保守估算: 按所有 `run_to` 上限和显示窗口相加, 会偏大.
- 实际运行仿真时间: 很多 `run_to` 会提前返回, 通常低于静态估算.

`na15` 优化后的关键收益来自减少重复 reset 和重复长路线, 而不是放宽题面性能窗口.
