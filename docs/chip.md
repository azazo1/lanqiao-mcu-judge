# 芯片仿真说明

本文记录一些和 STC15F2K60S2 相关, 且会直接影响仿真行为的实现约束与已知偏差.

## 指令时间与时钟分频

- STC15F2K60S2 是 `1T 8051`, 手册说明为 `1个时钟/机器周期`.
- 因而单条指令的实际执行时间, 取决于两部分:
  - 该指令本身需要多少个机器周期, 详见手册第 5 章指令系统, 其中 5.2 给出了各条指令的执行时间.
  - 当前系统时钟频率是多少.
- 手册对 `CLK_DIV` 寄存器的说明是: `系统时钟` 等于 `主时钟` 分频后的结果, 并且这个系统时钟会供给 CPU, 串口, SPI, 定时器, CCP/PWM/PCA, A/D 等模块.
- `CLK_DIV.CLKS[2:0]` 可将系统时钟设置为主时钟的 `1/2/4/8/16/32/64/128`.
- 因而从效果上说, 固件如果修改 `CLK_DIV.CLKS[2:0]`, 就是在成比例地拉长或缩短指令时间. 例如同一条 `1` 机器周期指令, 在 `SYSCLK = MCLK/1` 时执行一次需要 `1 / MCLK`, 在 `SYSCLK = MCLK/8` 时则需要 `8 / MCLK`.
- `MCKO_S1:MCKO_S0` 只是控制主时钟是否从 `MCLKO` 管脚对外输出, 以及输出时是否再除以 `1/2/4`, 它不会单独改变 CPU 指令执行速度.
- 对 STC15F2K60S2 当前供货的 C 版本, 手册还特别说明其主时钟只能来自内部 R/C 时钟. 后续升级型号 `STC15W2K60S2` 才支持外部输入时钟或外部晶体振荡.
- 当前代码里的 `approximate_instruction_ticks()` 位于 `src/chip.rs`, 它使用的是面向 STC15 仿真的近似机器周期表, 不是普通 8051 教材里常见的默认时序表.
- 这里和普通 8051 最容易混淆的点是: 很多通用 8051 资料默认按 `12T` 内核来讲, 而 STC15 是 `1T` 内核. 即使指令助记符相同, 在相同主时钟下, 实际 wall time 也会明显不同, 不能直接照抄普通 8051 的时间表来判断波形.
- 需要核对某条指令时间时, 优先查 STC15 手册第 5 章:
  - `5.2 完整指令集对照表(与传统8051对照)` 给出了每条指令的执行时间.
  - `5.3 传统8051单片机指令定义详解` 可以继续核对指令语义.
- 如果要对照当前仿真实现, 除了看 `src/chip.rs` 里的 `approximate_instruction_ticks()`, 还可以看同文件中的测试 `approximate_instruction_ticks_matches_stc15_common_gpio_ops()`.

## 串口中断与 `RETI` 返回

- STC15 的串口中断请求条件是 `RI` 或 `TI` 置位.
- `RI` 和 `TI` 都需要软件清零. CPU 响应串口中断时, 不会自动清掉这两个标志.
- 根据手册, `RETI` 返回之后, CPU 会先回到主程序执行 1 条指令, 然后才会再次响应新的同级中断请求.
- 这条规则对 `sample/uart` 很关键. 该样例的串口 ISR 只处理 `RI`, 发送路径则在 `putchar` 中轮询 `TI`, 然后执行 `TI = 0`.

## 已修复的历史偏差

- 旧版本仿真在 `RETI` 后会过早重新受理串口中断.
- 根因是外层调度和底层 `i8051::Cpu::step()` 都会在取指前先检查一次中断请求, 但底层并没有实现 `RETI` 后先执行 1 条主程序指令的延迟语义.
- 结果是主程序还没执行到清 `TI` 的那条指令, 就再次进入串口中断. 在波形里通常会看到连续的 `UART enter`, 看起来像 `TI` 没有复位.
- 这个现象与实机不一致. 实机上, `TI` 触发的中断次数应与实际发送完成的字节数一致, 不会因为同一个 `TI = 1` 连续多次进入中断.

## 当前实现约定

- 当前仿真固定按 `1T + 12MHz 主时钟` 建模, 并使用一份近似的指令耗时表来推进虚拟时间.
- 目前还没有把固件对 `CLK_DIV` 的动态写入接到 CPU 执行速度上. 也就是说, 真机上如果通过 `CLK_DIV.CLKS[2:0]` 改了系统时钟, 仿真中的指令时间现在不会同步变化.
- 当前仿真已经按手册修正为: `RETI` 之后先执行 1 条主程序指令, 然后才允许再次受理新的同级中断.
- 这 1 步只屏蔽取指前的中断轮询, 不屏蔽该指令自身对 `SCON`, `TI`, `RI` 等状态位的读取.
- 因而 `while (TI == 0); TI = 0;` 或 `JBC TI, ...` 这类典型发送收尾代码, 现在可以像实机一样及时清掉 `TI`.

## 排查建议

- 如果怀疑串口中断重入异常, 优先同时查看 `event.cpu`, `event.uart1`, `uart1.ti`, `uart1.ri`, `uart2.ti`, `uart2.ri`.
- 如果波形里出现大量连续的 `UART enter`, 需要先区分是固件真的没有清标志, 还是仿真时序偏差导致主程序来不及执行清标志指令.
